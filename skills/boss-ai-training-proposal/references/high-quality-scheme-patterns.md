# High-Quality Scheme Patterns

Use this reference when the user wants a training scheme that is more actionable than a basic outline. This is especially useful for transformation projects, platform upgrades, business-model workshops, and practical operating training.

## What makes a stronger scheme

A weak scheme usually stops at:

- topic list
- broad day split
- generic theory

A stronger scheme adds:

- training theme
- training audience
- training duration and practical ratio
- concrete core goals
- detailed module flow by time
- interactive design
- visible outputs
- risk control
- phased rollout suggestions
- pre/in/post training support

## Recommended opening block

Prefer opening with:

- `培训主题`
- `培训对象`
- `培训时长`
- `核心目标`

Example shape:

```md
- 培训主题：<一句话主题，清楚说明转型方向或能力目标>
- 培训对象：<岗位/部门/人数范围>
- 培训时长：<X天，每天X小时，合计X小时；实操模块占比X%>
- 核心目标：
  1. <目标1>
  2. <目标2>
  3. <目标3>
  4. <目标4>
```

## Better two-day or three-day schedule design

Do not stop at `第一天上午讲趋势，下午讲案例`.

Instead, make the schedule progress by logic:

1. 认知统一
2. 标杆拆解
3. 本地适配
4. 实操打法
5. 风险控制
6. 落地试点

For each day, split by exact time ranges and visible modules.

## Module design pattern

For each module, include at least:

- `核心目标`
- `主要内容`
- `互动形式`

When useful, also include:

- `关键产出`
- `配套工具`
- `案例素材`

Recommended writing pattern:

```md
模块一：<模块名称>（1.5小时）
- 核心目标：<一句话>
- 主要内容：
  1. <内容1>
  2. <内容2>
  3. <内容3>
- 互动形式：<工作坊 / 演练 / 分组讨论 / 拆解报告 / 现场点评>
- 关键产出：<模板 / 方案 / 清单 / 路径图>
```

## Interaction patterns

High-quality schemes should not be pure lecture. Choose fitting interactions:

- 痛点共创工作坊
- 案例分析会
- 分组拆解报告
- 方案共创
- 实操演练
- 风险排查游戏
- 分组制定试点推广计划

Match the interaction to the module:

- trend and diagnosis -> 案例分析 / 共创
- benchmark teardown -> 分组拆解
- operating SOP -> 实操演练
- compliance and after-sales -> 风险排查
- rollout plan -> 分组计划汇报

## Stronger transformation-scheme checklist

If the request is about business transformation, platform upgrade, or new commercial model, try to cover:

- 项目背景与业务痛点
- 宏观趋势与行业机遇
- 标杆平台底层逻辑
- 本地化适配方案
- 实操运营策略
- 风险控制体系
- 商业模式上限与可行性预判
- 分阶段落地行动计划
- 工具包交付

Not every scheme needs all nine, but a high-quality transformation scheme usually covers most of them.

## Pre / in / post support

When the scheme is client-facing, add a dedicated保障 section:

- `课前`: 预习资料、功能材料、调研问题、案例收集
- `课中`: 讲义、二维码、讲师配置、录像回看、现场答疑
- `课后`: 答疑会、复盘支持、工具包、更新版模板

This makes the方案 feel more complete and more commercially credible.

## Apply this carefully

- Borrow the structure, not the exact industry wording.
- Keep industry-specific examples relevant to the customer.
- Do not invent metrics or targets unless they are inferable or clearly labeled as assumptions.
- If the company already has strong foundations, reduce basic theory and increase practical design and decision content.
