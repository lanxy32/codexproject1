# Formal Proposal Template

Use this reference when the user wants a polished client-facing proposal based on the preferred pattern:

- `3天集中内训`
- `8周线上深化`
- `1个月专属咨询`

Treat this as a high-quality commercial reference for customers with weak AI foundation and multiple departments. Do not force this structure when the user asks for a different arrangement.

## Stronger opening fields

When the proposal is more教案化 or运营导向, open with:

- 培训主题
- 培训对象
- 培训时长
- 核心目标

If helpful, mention:

- 总课时
- 实操占比
- 参训人数范围

## Recommended section order

1. 项目概述
2. 项目目标
3. 整体服务流程
4. 3天集中内训大纲
5. 3天内训内容展开
6. 8周线上深化学习大纲
7. 配套服务内容
8. 项目交付成果
9. 客户配合事项
10. 方案价值总结

## Recommended wording

Use language like:

- `集中打基础 + 持续练应用 + 陪跑保落地`
- `帮助企业从AI认知进入岗位应用与智能体梳理阶段`
- `不是一次性培训，而是AI应用建设的启动项目`
- `通过持续学习机制提升培训成果转化率`

Avoid language like:

- `三天完成企业AI转型`
- `所有部门立即实现自动化`
- `培训后无需继续投入`

## Recommended service-flow table

| 阶段 | 服务内容 | 目标 |
|---|---|---|
| 需求调研阶段 | 签订合同后，通过问卷和部门访谈梳理真实痛点、典型流程与应用期待 | 调整培训重点，提高匹配度 |
| 集中内训阶段 | 3天现场培训，完成AI认知、场景拆解、智能体清单盘点、部门实操 | 完成第一轮统一认知和上手 |
| 持续学习阶段 | 8周线上深化学习，每周1次，每次2小时 | 帮助员工从听懂到做熟 |
| 后续服务阶段 | 1个月专属线上咨询 | 解决应用与落地过程中的实际问题 |

## Recommended 3-day table

| 天数 | 培训对象 | 培训模块 | 培训内容 | 关键产出 |
|---|---|---|---|---|
| 第一天 | 全部门 | AI认知统一 + 需求梳理 | 工具认知、提示词基础、场景盘点、智能体清单梳理 | 部门应用清单初稿 |
| 第二天 | 内容、直播、设计相关部门 | 专项实操 | 文案、设计、拍剪、切片、直播运营、推流场景训练 | 内容链路模板与智能体候选清单 |
| 第三天 | 客服、采购、仓储、人事等部门 | 专项实操 + 落地复盘 | 客服、供应链、人事应用训练与复盘 | 支持链路应用清单与后续优先级 |

For a more detailed教案 version, prefer:

| 序号 | 天数 | 时间 | 培训模块 | 课程主题 | 课程详细内容 | 关键产出 |
|---|---|---|---|---|---|---|
| 1 | 第一天 | 09:30-10:20 | AI认知统一 | AI基础认知 | 行业趋势、工具认知、应用边界 | 统一认知 |

For a higher-quality实战版, describe each module underneath the table with:

- 核心目标
- 主要内容
- 互动形式
- 关键产出

## Recommended weekly-learning table

| 周次 | 学习主题 | 重点部门 | 学习目标 | 作业/产出 |
|---|---|---|---|---|
| 第1周 | 提示词强化与统一规范 | 全部门 | 巩固基础，统一输入输出方法 | 部门提示词模板 |

| 第4周 | 客服场景深化 | 客服部 | FAQ与标准回复体系化 | 客服模板 |
| 第5周 | 采购与仓储场景深化 | 采购部、仓储部 | 汇总、比价、异常与SOP优化 | 采购/仓储模板 |

## Use rules

- If the user wants a formal proposal, use this as a reference pattern rather than a fixed shell.
- If the user gives a company name and department list, you may fill them into this structure or simplify it.
- If the user wants only an outline, keep the strongest sections and trim the rest.
- If the user references screenshots, keep tables central and move prose to the top and bottom only.
- If the requested service周期 is not `3天 + 8周 + 1个月`, adapt the timeline freely.
- If the user asks for课程安排 or教案, default to exact session times rather than `上午/下午`.
- If the user wants a higher-quality方案, add module-level `核心目标 / 主要内容 / 互动形式 / 关键产出`.
